01 — Decisión

Justificación de la Metodología

"Para este proyecto se ha seleccionado la metodología Scrum debido a que la naturaleza de sus requisitos es cambiante y evolutiva — múltiples módulos Android (autenticación, boletos, streaming, biografía y administración) se desarrollan iterativamente — lo que exige un enfoque que priorice la flexibilidad y las entregas parciales. Esta elección mitiga el riesgo tecnológico al adaptarse a un nivel de incertidumbre alto (integración de Retrofit, ExoPlayer y pasarela de pago), se alinea con la disponibilidad del cliente para recibir feedback de forma continua y optimiza la dinámica de un equipo de 3 integrantes mediante el uso de GitHub Projects para la gestión visual de tareas y roles."
🔄

Requisitos Evolutivos

La integración de streaming, pagos y contenido multimedia genera descubrimientos técnicos que Scrum absorbe naturalmente en cada sprint.

⚠️

Incertidumbre Alta

Tecnologías nuevas como ExoPlayer HLS y pasarela de pago requieren iteración rápida y retroalimentación continua del cliente.

🤝

Feedback Continuo

El cliente del festival puede validar cada módulo (boletos, streaming, biografía) al final de cada sprint de 2 semanas.

👥

Equipo Pequeño

Con 3 integrantes, Scrum ofrece estructura suficiente sin la overhead burocrática de metodologías más pesadas.

02 — Marco de Trabajo

Estructura del Proceso Scrum

  • Product Backlog — Lista priorizada de todas las Historias de Usuario (HU-01 a HU-05). El Product Owner refina y reprioriza entre sprints según el feedback recibido.
  • Sprint Planning — Al inicio de cada sprint (2 semanas), el equipo selecciona las historias a completar y define las tareas técnicas en Android Studio / Kotlin.
  • Daily Standup — Reunión diaria de 15 minutos: ¿qué hice ayer?, ¿qué haré hoy?, ¿hay algún bloqueo? Coordinación del equipo de 3 personas.
  • Sprint Review — Demo funcional del módulo Android completado al cliente/docente al final de cada sprint. El incremento debe ser usable y desplegable.
  • Sprint Retrospective — Análisis interno del equipo: qué salió bien, qué mejorar técnicamente y en la dinámica de trabajo.

Sprint 0 Setup del proyecto Repo GitHub, estructura MVVM, GitHub Pages con PID + SAD.

Sprint 6 Cierre y QA Pruebas unitarias ≥80%, ajustes de UX y entrega final.

03 — Gestión

Herramienta de Gestión — GitHub Projects

El equipo utilizará GitHub Projects como tablero Kanban para la gestión visual de tareas, vinculando cada issue directamente con los commits y pull requests del repositorio.

📋

Columnas del Tablero

Backlog → En Progreso → En Revisión → Completado

🔗

Integración con Git

Cada issue se cierra automáticamente al hacer merge del PR correspondiente en la rama main.

🏷️

Etiquetas

Labels por módulo: auth, boletos, streaming, biografia, admin, bug.
